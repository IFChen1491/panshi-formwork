磐石 SaaS 不該被外包成別人的資料庫

本次決策不是「外包或內製」二選一，而是要切出主權邊界：營運速度可以外借，資料主權、派工邏輯、3D 配模演算法與品牌敘事必須長在磐石自己身上。

01｜先結論

採用混合路線：買速度，不賣主權

套裝 ERP 可以先跑營運，讓現場流程、打卡、進銷存、工人坪效先有資料回流。內部團隊負責第一線介面、3D 配模操作、ERP 經驗內化。洞察團隊只承接底層架構、演算法、資安、補助文件與後端規格，不取得磐石主資料庫與派工主控權。

AI 傾向：C 路線，外部做底層與補助，內部掌產品與資料，ERP 只當過渡驗證工具。 建議定案方向
A｜全外包 SaaS

快，但長期危險

短期可拿補助、規格書漂亮，但資料庫、迭代節奏與演算法會被外部牽動。

速度
主權
B｜全內部自建

健康，但太慢

最符合長期演化，但第一線營運、補助文件與資安架構會拖慢啟動速度。

速度
主權
C｜混合路線

最符合磐石階段

ERP 買速度，內部掌產品，洞察補底層，補助文件同步成形。

速度
主權

02｜路徑建議

四條路徑，只有一條適合現在的磐石

主推：ERP 先跑，磐石 SaaS 內化，洞察補底層

適用情境

需要快速營運、同步準備補助、又不能讓資料主權外流。

關鍵條件

主資料庫歸磐石，外部只拿 API 與同步副本，補助文件不可把主體寫成供應商。

第一步

Apple 本週定義內部 SaaS 模組，誠致列補助文件清單，洞察後續補架構圖。

ERP 過渡：15 萬買一個現場驗證場

適用情境

先確認公司流程是否真的能跑，不急著把全部 SaaS 一次寫完。

關鍵條件

必須可匯出資料、可定義欄位、可接 API，否則 ERP 會變成第二個資料孤島。

第一步

用一個案場或一個工種試跑，30 天後決定哪些流程移植到內部 SaaS。

內部自建：長期正確，但不能孤軍慢跑

適用情境

磐石要把 3D 配模、模主制度、派工權、施工回饋變成長期護城河。

關鍵條件

需要明確 90 天 MVP，不可陷入「整套 SaaS 一次到位」。

第一步

只做三件事：模板 ID、師傅回饋、配模施工圖與數量表回流。

全外包：只有在合約主權完整時才可考慮

適用情境

外部公司願意明確交付原始碼、資料匯出、API、部署文件與演算法成果授權。

關鍵條件

維護費、客製費、資料所有權、品牌案例權都要白紙黑字。

第一步

未簽主權條款前，不進入正式開發，只允許做補助規格與技術顧問。

03｜多 Agent 協作分工

把外包問題拆成八個可控責任面

IF 決策 Agent

主權裁決

決定外部合作邊界，確保數據、派工、財務、品牌不失守。

SaaS 架構 Agent

系統藍圖

切分內部 SaaS、ERP、洞察後端與 GCP/PostgreSQL 的責任。

數據主權 Agent

SSOT 守門

定義 NoSQL 與 PostgreSQL 的同步、備份、權限與主資料庫。

現場 UX Agent

師傅介面

設計師傅看圖、回報、拍照、缺陷碼與 3D 配模施工圖回饋。

ERP 整合 Agent

快速營運

用套裝 ERP 驗證流程，盤點欄位、匯出能力與內化價值。

補助 PM Agent

申請控管

整理文件清單、送件主題、補助時程與審查合規性。

洞察協作 Agent

底層技術

提供演算法、資安、API、雲端架構與補助用技術規格。

財務 ROI Agent

成本效益

比較補助、外包費、內部延誤成本與長期維護費。

04｜工作流

從 CAD 到師傅回饋，必須形成資料閉環

01

CAD 抓取

由內部 3D 配模能力與工程圖資料建立初始模型。

02

AI 疊代

洞察可協助演算法，但資料標準與成果歸磐石。

03

施工圖與數量表

輸出師傅版配模施工圖、模板數量與模板 ID。

04

現場回饋

師傅依圖施工，回傳照片、缺陷碼、耗時與坪效。

05

ERP / SaaS 回流

工人打卡、模板 ID、施工進度與成本回到 SSOT。

05｜風險與紅線

本案最大風險不是花錢，是資料被拆散

1NoSQL / PostgreSQL 分裂

若沒有主資料庫與 ETL 規格，雲端、地端、ERP、SaaS 會各自說真話。控制方式：定義磐石 SSOT，外部資料庫只能是同步副本。

2補助牽著產品走

若為了補助題目過度客製，會把產品主線帶偏。控制方式：補助文件服務於磐石路線，不讓補助變成產品總設計師。

3派工權外移

ERP 或外部演算法若開始替平台決定派工，就踩到主權紅線。控制方式：AI 可建議，派工權留在磐石制度。

4品牌敘事被供應商拿走

補助案、Demo、架構圖若把主體寫成技術供應商，磐石會失去案例敘事權。控制方式：所有對外材料主詞是磐石系統模板平台。

06｜時程與責任

先把本週的邊界畫清楚，才讓外部開始報價

24 小時內

誠致顧問陳志

列出補助所需文件清單：架構圖、模型說明、規格書、預算表。

本週內

陳總 + Apple

完成內部會議，盤點 SaaS 前後端、ERP 試用範圍與不可外包項目。

7 天內

Apple + 洞察技術長

定義 NoSQL / PostgreSQL 同步、API、備份與權限規格。

14 天內

洞察團隊

提供後端系統規劃、報價單、補助用系統企劃書與底層架構圖。

30 天內

補助 PM Agent

決定第一案送件題目，建議優先以 3D 模板數量與智慧施工 RFID 整合切入。

60 至 90 天

內部團隊 + 現場

完成 ERP 試營運回顧，決定哪些流程內化到磐石 SaaS。

07｜Gate 與閉環

每一輪都要有可交付物，不靠會議感覺推進

A主權 Gate

外部不得掌握主資料庫、派工規則、品牌敘事與不可替代的核心演算法。

BERP Gate

套裝 ERP 必須可匯出、可接 API、可定義模板 ID 與工人紀錄欄位。

C補助 Gate

架構圖、模型說明、規格書、報價單、可行性說明齊備才進入送件。

D內化 Gate

ERP 試營運資料必須轉成磐石 SaaS 欄位、流程與產品待辦。

強制收斂主張

採用混合路線

短期 ERP 買營運速度，中期內部 SaaS 掌主權，外部洞察承接底層演算法、資安與補助文件。

唯一反轉條件

全外包必須先簽主權條款

除非外部公司承諾原始碼、全量資料匯出、部署文件、API、演算法成果使用權與合理維護費，否則不建議全外包。

Closing Mission

磐石要建立的是營建業 OS，不是買一套表單系統

補助是燃料，ERP 是過渡工具，洞察是底層協力者。真正的主體是磐石自己的資料、流程、現場回饋、模主制度與可持續演化的 SaaS 核心。
